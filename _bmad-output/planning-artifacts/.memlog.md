- (decision) Product thesis: Hexalith.Folders is the tenant-scoped workspace control plane for production agentic file work, replacing caller-owned filesystem, Git, credential, cleanup, and recovery orchestration rather than becoming a generic file manager or Git UI.
- (decision) Canonical MVP job: prove an AI agent can pass provider readiness, create or bind a repository-backed folder, prepare and lock a workspace, add/change/remove files, commit, query context and status, and inspect metadata-only audit.
- (decision) MVP scope is repository-backed first; local-only folders, local-to-Git promotion, brownfield adoption, repair automation, and multi-agent simultaneous writes are deferred so tenant isolation, locking, persistence, and failure visibility are proven first.
- (decision) REST v1 is the canonical external contract; CLI, MCP, and SDK must preserve equivalent lifecycle, authorization, idempotency, errors, operation identity, status, and audit semantics, while the console remains a projection.
- (decision) Tenant isolation is a zero-tolerance invariant: unauthorized and cross-tenant operations must be denied before file, workspace, credential, repository, lock, commit, provider, audit, context, or hidden-resource existence can be accessed or inferred.
- (decision) Authorization combines authoritative tenant context with folder-specific ACL verbs and evaluates principal, delegated actor, provider, repository, folder, workspace, and task scope for every operation.
- (decision) Concurrency uses one active mutation writer per tenant, folder, repository binding, and task workspace scope; governed mutations and commits require a valid lock, while authorized context and status reads do not, and takeover must never be silent.
- (decision) Idempotency is required for workspace preparation, lock acquisition, file mutation, commit, and cleanup requests; equivalent replays return the same logical result, conflicting intent is rejected, and unknown provider outcomes require reconciliation before retry.
- (decision) The control plane is metadata-only: file contents, diffs, generated context payloads, provider tokens, credential values, secrets, and unauthorized existence stay out of events, logs, traces, metrics, projections, audit, diagnostics, errors, and console views; working content remains outside EventStore.
- (decision) MVP provider scope is GitHub and Forgejo, each gated by readiness validation, explicit capability reporting, and provider contract evidence rather than assumed API compatibility.
- (decision) The MVP operations console is read-only and projection-based, showing readiness, binding, lock, dirty, commit, failure, credential-reference, provider, sync, audit, and timeline trust signals without mutation, repair, file editing, or content browsing.
- (decision) Interrupted and failed work must remain inspectable through explicit workspace, lock, operation, retryability, and reconciliation states; MVP exposes safe blocked or dirty state and does not silently repair, discard, release, or commit work.
- (decision) Tenant administrators can grant and revoke folder access, inspect effective permissions, and archive folders; archived folders reject mutation while retaining policy-bound metadata-only status and audit evidence.
- (decision) MVP acceptance prioritizes a complete cross-surface repository workflow, zero cross-tenant leaks, GitHub and Forgejo contract passes, deterministic lock and retry behavior, and sufficient metadata-only evidence to reconstruct successful and failed tasks.
- (decision) FR58 currently means authorized metadata-token recall through the Memories index: results are security-trimmed to tenant/folder/workspace, hydrated from authoritative Folders reads, and redact raw paths, snippets, source URIs, bodies, and hidden-resource existence; real-content recall remains a C9-gated follow-up.
