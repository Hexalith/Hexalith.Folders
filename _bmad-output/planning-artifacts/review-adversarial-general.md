# Adversarial Review — Hexalith.Folders PRD

- The document has no trustworthy current-baseline marker: frontmatter still says `status: complete`, `completedAt: 2026-05-07`, and `lastEdited: 2026-05-07`, while the body contains decisions and implementation references from late May through July. The listed input documents also omit the architecture and project context that now materially qualify the product contract, so a downstream reader cannot tell which assertions were reviewed together.
- The PRD still classifies the product as `greenfield` even though it cites implemented epics, reopened stories, live release-readiness closure, approved exit criteria, and existing Contract Spine behavior. That historical label now misstates the planning context and encourages downstream work to ignore brownfield compatibility, migration, and regression constraints.
- C3 and C4 remain `TBD by architecture review` even though the authoritative governance state marks both approved: C3 has PM and Legal approval with binding per-data-class retention periods, and C4 has PM-approved numeric input and response limits. This directly violates the PRD's own instruction that approved targets be referenced back into the PRD and leaves security, deletion, idempotency TTL, query bounds, and acceptance decisions falsely open.
- MVP acceptance requires provider callbacks and webhooks to be tenant-bound, replay-safe, and duplicate-delivery tested, while the architecture explicitly declares no webhook ingestion in MVP and requires webhook routes to return `404`. The PRD does not distinguish incoming provider webhooks, outbound callbacks, or generic asynchronous worker retries, so teams can either omit stated MVP evidence or accidentally introduce a forbidden ingress surface.
- The PRD requires the operations console to consume projections only and prohibits direct event mirrors, while the architecture adds an MVP ACL-checked `/_admin/incident-stream` view over authoritative event streams. If the incident path is intended product behavior, the PRD lacks its actor, permission, redaction, ordering/freshness warning, audit, availability, and acceptance requirements; if it is not intended, the architecture has expanded MVP scope beyond the product contract.
- Contract authority is internally unstable: the PRD calls REST the canonical external contract, makes the SDK an adapter, and elsewhere allows REST plus one SDK to survive a scope cut, while current project governance calls the SDK the typed canonical client and OpenAPI the Contract Spine. The document does not cleanly separate authoritative schema, server transport, generated client, and adapter semantics, leaving versioning and parity disputes without a product-level tie-breaker.
- SDK scope is not decidable: it is described as required for "MVP or near-MVP," the MVP success and parity evidence require it, and the resource fallback preserves it while allowing CLI and MCP to slip, even though CLI and MCP are repeatedly called must-have MVP surfaces. A release team cannot derive one unambiguous minimum surface set or acceptance gate from these statements.
- Binding an existing repository is part of the canonical MVP workflow and FR19, but brownfield folder or repository adoption is also declared post-MVP and a brownfield migration wizard is an explicit non-goal. "Where supported" does not define whether MVP binding permits empty repositories, repositories with history, pre-existing working trees, arbitrary default branches, or only repositories created by Folders, so the central repository lifecycle cannot be scoped consistently.
- Tenant-administration scope is disconnected from MVP scope: Journey 9 and FR4–FR14 require access administration, effective-permissions inspection, archive, and retained evidence, but the MVP feature set and acceptance evidence omit those capabilities. FR5 specifies grant but not revoke despite the journey and summary promising both, and FR13–FR14 do not carry the journey's required denial of all archived-folder mutations or its retention-bound visibility behavior.
- FR58 promises that clients can "search the content," then redefines fulfillment as metadata-token recall with no body text, snippets, raw paths, or source URIs. Those are materially different user capabilities; treating the metadata surrogate as satisfaction of content search makes the requirement semantically false and allows a weak implementation to pass an acceptance test written from the headline.
- FR58 is not integrated into the rest of the product case: there is no corresponding user journey, success measure, MVP feature, acceptance scenario, error behavior, freshness expectation, or explicit release phase for the Memories-backed search facade. Its scope note instead embeds Epic 10 story numbers, worker/bridge mechanisms, approval-gate names, and incomplete implementation status, turning the chain-top PRD into a volatile delivery ledger while still failing to state a testable product outcome.
- The workspace-state contract mixes independent dimensions into one purported state model: `locked` is concurrency ownership, `dirty` is working-copy condition, `inaccessible` is an authorization or availability result, and `ready`/`committed`/`failed` are lifecycle outcomes. Additional lists introduce `Pending`/`InProgress`/`Succeeded`, `ProviderReady`/`RepositoryBound`/`FilesChanged`, and `requested`/`preparing`/`reconciliation_required` without defining composition or precedence, so FR45 and cross-surface state parity have no single verifiable meaning.
- Zero-tolerance authorization is underspecified for long-running and asynchronous work. The PRD requires authorization before access but does not state what happens when tenant membership, folder ACL, delegated-agent authority, provider binding, or credential permission is revoked after prepare or lock and before a worker mutates files or commits. Without a required freshness/revalidation outcome and fail-closed terminal state, "authorized at request receipt" can silently become unauthorized execution.
- The platform-operator authority model is missing. Journeys allow an operator to search for a tenant and folder and view credential-reference, provider, lock, failure, and audit information, while the tenant-isolation invariant forbids visibility or inference across tenants. The PRD does not define which operator role may cross tenant boundaries, whether tenant consent or need-to-know scope applies, what fields are redacted, or how such privileged reads are audited and made distinguishable from tenant-member access.
- The lock invariant does not identify the authoritative collision key. "One active writer per tenant, folder, repository binding, and task workspace scope" can be read as a composite that permits two folders or branch aliases to mutate the same underlying repository/ref concurrently. The PRD requires duplicate-binding rejection but does not define repository/ref identity, alias detection, or whether locks serialize per folder, repository, branch, working copy, or some combination, leaving wrong-repository mutation and mixed-commit prevention unverifiable.
- "Commit" is never defined as a local Git commit, a provider-side commit, or a commit that has been pushed to the bound remote and confirmed durable. The product promise is provider-backed persistence, but no push/sync operation or remote-durability acceptance outcome is named; a local SHA and clean working copy could therefore satisfy the written demo while losing the advertised durability after workspace cleanup.
- The required error `details` field has no normative safe-field schema, size bound, per-code visibility rule, or equivalence rule for unauthorized versus nonexistent resources. Given the zero-tolerance non-enumeration promise, an unconstrained `details` bag is a direct leakage and cross-surface-drift surface even if the top-level category and message are sanitized.
- The claimed `100%` adapter-parity, ACL-matrix, and command/query coverage gates have no stable denominator in the PRD. The DTO list is explicitly only "at least" a partial list and omits administration, archive, effective-permission, Memories search, and several diagnostic operations, while no normative ACL matrix is included or referenced. A test suite can report 100% by choosing a convenient inventory rather than covering all FRs.
- User-visible file policy remains deferred as if it were only an implementation choice. The PRD requires traversal, symlink, Unicode normalization, case collision, encoding, binary, large-file, and include/exclude behavior to be enforced, but it does not state the allowed outcomes or identify a now-authoritative policy artifact. Different adapters or providers can therefore reject, normalize, follow, truncate, or accept the same request differently while each claims compliance.
- The success section is mostly a feature-existence checklist rather than measurable product success: task completion rate is tracked without a target or observation window, three- and twelve-month outcomes have no adoption or reliability threshold, and there are no counter-metrics for security denials, lock contention, dirty terminal states, provider failure, latency, or operator burden. The release can technically satisfy every listed outcome without demonstrating that the product is reliable or adopted.
- Sensitive metadata policy is left open despite approved downstream governance. The PRD says paths, commit messages, repository names, branch names, and provider diagnostics must be classified and protected "where appropriate," while current C9 policy assigns a default tenant-sensitive tier and a confidential override behavior. Leaving the default and permitted views out of the product contract makes audit reconstruction, console diagnostics, FR58 redaction, and cross-tenant operator behavior depend on undocumented interpretation.
- Major product decisions have no durable rationale trail: there is no `.memlog.md` or addendum, and the edit history does not recover the decisions behind removing local-mode MVP scope, adding tenant-administrator lifecycle requirements, adding FR58, or accepting the architecture's later boundary changes. Future update work cannot reliably distinguish intentional product decisions from prose drift or implementation-driven scope insertion.
