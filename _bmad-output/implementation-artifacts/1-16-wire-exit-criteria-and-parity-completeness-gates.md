# Story 1.16: Wire exit-criteria and parity completeness gates

Status: ready-for-dev

Created: 2026-05-14

## Story

As a maintainer,
I want governance and completeness gates wired into CI,
so that missing release evidence, unstable idempotency encoding, stale examples, or unsafe tenant cache keys block implementation.

## Acceptance Criteria

1. Given the repository contains C0-C13 exit-criteria evidence artifacts or explicit reference-pending records, when the governance gate runs, then every C0-C13 row maps to an artifact path, owner, status, verification command, and result summary, and missing or malformed evidence fails closed.
2. Given `docs/exit-criteria/` contains approved workshop outputs and placeholders, when the exit-criteria presence gate runs, then it distinguishes approved evidence, bounded `reference_pending` decisions, and invalid placeholders without treating unresolved C3/C4/C6/S-2 policy as silently complete.
3. Given `tests/fixtures/idempotency-encoding-corpus.json` and its schema exist, when the idempotency-encoding equivalence gate runs, then corpus cases validate against the schema and every case is consumed by focused helper or parser-policy tests that preserve null-versus-omitted, ordering, Unicode, percent-decoding, casing, malformed-key, and duplicate-key expectations.
4. Given Story 1.12 owns generated `ComputeIdempotencyHash()` helper behavior, when this story adds gates, then it verifies helper/corpus coverage and deterministic local commands without implementing new SDK generation policy or editing generated client output.
5. Given architecture pattern examples are developer-facing contract, when the pattern-example compilation gate runs, then code snippets or extracted examples in architecture, contract docs, and gate documentation compile or fail with a bounded prerequisite-drift diagnostic.
6. Given cache keys can leak or mix tenants, when the tenant-prefixed cache-key lint runs, then in-process, Dapr state, Redis, distributed-cache, generated-helper, test-fixture, and workflow cache keys present in the repository are rejected unless they are tenant-scoped, explicitly non-tenant, or documented synthetic examples.
7. Given `tests/fixtures/parity-contract.yaml` may be generated by Story 1.13, when the parity completeness gate runs, then every current OpenAPI operation appears exactly once in the parity oracle and every parity row references an existing operation ID, adapter expectation, operation family, idempotency rule, and read-consistency classification.
8. Given Story 1.14 owns Contract Spine drift, generated-client golden-file consistency, parity schema validation, and CI workflow wiring, when this story adds completeness gates, then it extends or reuses that lane without duplicating restore/build/test steps or reimplementing server-vs-spine drift detection.
9. Given Story 1.15 owns safety invariant gates, when this story is complete, then it does not implement sentinel redaction, forbidden-field scanning, output-channel leakage checks, or safety corpus policy outside cache-key and diagnostic metadata needed by its own gates.
10. Given some prerequisites may still be absent, when any gate cannot evaluate because a prior story has not produced its owned artifact or command, then it records exact prerequisite drift and fails closed instead of generating placeholder success, inventing parity rows, or broadening scope.
11. Given CI diagnostics are visible outside the local developer machine, when a governance or completeness gate fails, then logs include only gate names, repository-relative paths, criterion IDs, operation IDs, schema pointers, sample IDs, bounded categories, and safe hashes; logs never echo raw payloads, file contents, diffs, provider tokens, credentials, tenant data, local absolute paths, production URLs, cache key values, or unauthorized-resource hints.
12. Given gates must be locally reproducible, when implementation is complete, then documented repository-root commands run the same exit-criteria, idempotency-encoding, pattern-example, cache-key, and parity-completeness checks without Aspire, Dapr sidecars, Keycloak, Redis, GitHub, Forgejo, provider credentials, tenant seed data, production secrets, network calls, or nested submodule initialization.
13. Given active Story 1.11 through Story 1.15 work may still be dirty or reference-pending, when implementation starts, then the developer inspects current OpenAPI, exit-criteria docs, idempotency fixtures, parity fixtures, contract docs, workflow files, and ready story artifacts before assuming prerequisites are implemented.
14. Given this story wires governance and completeness gates only, when implementation is complete, then it does not add runtime REST handlers, EventStore commands, domain aggregate behavior, provider adapters, Git or filesystem side effects outside deterministic tests/tools, SDK generation policy, parity-oracle derivation, CLI commands, MCP tools, UI pages, Dapr policy conformance, provider live-drift jobs, release publishing, safety invariant scanning, or nested-submodule initialization.

## Dev Notes

### Scope Boundaries

- This story wires CI and local gate entry points for exit-criteria presence, idempotency-encoding coverage, pattern-example compilation, tenant-prefixed cache-key lint, and parity completeness.
- Allowed implementation areas are:

```text
.github/workflows/
docs/contract/governance-and-completeness-ci-gates.md
docs/exit-criteria/
tests/fixtures/idempotency-encoding-corpus.json
tests/fixtures/idempotency-encoding-corpus.schema.json
tests/fixtures/parity-contract.yaml
tests/Hexalith.Folders.Contracts.Tests/
tests/Hexalith.Folders.Client.Tests/
tests/Hexalith.Folders.Testing.Tests/
tests/tools/
```

- Equivalent file names are acceptable when they preserve the same ownership boundaries.
- Do not implement runtime behavior, Contract Spine operation groups, SDK generation policy, parity-oracle derivation, safety-invariant scanning, provider drift, Dapr policy conformance, CLI/MCP/UI work, release publishing, or nested-submodule initialization.
- This story may inspect generated or checked-in artifacts but must not hand-edit generated SDK output or generated parity rows.

- `.github/workflows/` is currently absent unless Story 1.14 creates it before implementation begins.
- `docs/exit-criteria/` currently contains `_template.md`, `c3-retention.md`, `c4-input-limits.md`, `c6-transition-matrix-mapping.md`, and `s2-oidc-validation.md`.

- `docs/contract/idempotency-and-parity-rules.md` is the authoritative idempotency and adapter parity rules artifact.

- Active Story 1.11 review work may leave dirty OpenAPI, contract docs, and contract-test files. This story must inspect current state but must not absorb Story 1.11 implementation scope.

### Gate Requirements

- Exit-criteria checks must derive C0-C13 from architecture rather than a duplicated hard-coded checklist unless the hard-coded list is generated from an approved source.
- Evidence records must distinguish approved, failed, reference-pending, and placeholder states. Generic placeholder evidence must fail once the gate is active.
- Idempotency encoding checks must validate both fixture schema and test consumption. Fixture existence alone is not enough.
- Pattern-example compilation must be opt-in for examples marked as compilable; examples that illustrate YAML, JSON, workflow snippets, or pseudocode need explicit non-compilable classification.
- Cache-key lint must reason about tenant scope, not just string prefixes. Safe exceptions need explicit classification and must not include tenant, provider, or environment values.
- Parity completeness must compare current OpenAPI operations to generated parity rows. It must not rely on stale operation inventories in story text.
- Gate diagnostics must be safe by construction. Prefer criterion IDs, operation IDs, sample IDs, category names, schema pointers, and content hashes over raw values.

### Latest Technical Notes

- Current `actions/setup-dotnet` guidance shows `actions/checkout@v6` with `actions/setup-dotnet@v5`, and the action can read SDK versions through `global-json-file`; use the repository `global.json` rather than relying on runner-installed SDK defaults. Source: https://github.com/actions/setup-dotnet
- `actions/setup-dotnet` NuGet caching is optional and off by default; when enabled, it expects NuGet lock files or `cache-dependency-path`. Defer caching unless the implementation can prove lock-file support and safe cache-key classification. Source: https://github.com/actions/setup-dotnet
- GitHub Actions workflow files are YAML files with jobs and steps; keep gate commands visible as `run` steps or reusable local scripts so developers can reproduce failures outside GitHub Actions. Source: https://docs.github.com/en/actions/reference/workflows-and-actions/workflow-syntax
- Microsoft .NET CLI documentation states `dotnet build` and `dotnet test` perform implicit restore, but this repository should keep explicit `dotnet restore Hexalith.Folders.slnx` first in CI to isolate package resolution and lock-file failures. Sources: https://learn.microsoft.com/en-us/dotnet/core/tools/dotnet-build and https://learn.microsoft.com/en-us/dotnet/core/tools/dotnet-test

### Testing Guidance

- Tests must run offline and must not require Aspire, Dapr sidecars, Keycloak, Redis, GitHub, Forgejo, provider credentials, tenant seed data, production secrets, network calls, or initialized nested submodules.
- Prefer structured parsing: YAML for OpenAPI/parity/exit evidence, JSON for fixture schemas and corpora, and Roslyn or a small compile harness for C# snippets where practical.
- Good negative cases: missing C13 evidence row, generic `PLACEHOLDER` evidence after a gate is active, an idempotency corpus case with no consumer test, a helper test that collapses null and omitted, a compilable example with an API drift error, a tenant data cache key without tenant scope, a parity row for a removed operation, duplicate parity rows, and failure diagnostics that try to echo raw values.
- Good positive cases: bounded reference-pending decisions name the criterion and owner, every idempotency corpus case maps to a consumer, examples marked documentation-only are skipped with evidence, non-tenant tool caches are explicitly classified, and parity completeness derives current operations from OpenAPI.
- Keep assertions precise and scoped to repository-owned artifacts. Do not add broad secret scanning or release validation claims that belong to Story 1.15 or later production-hardening stories.
