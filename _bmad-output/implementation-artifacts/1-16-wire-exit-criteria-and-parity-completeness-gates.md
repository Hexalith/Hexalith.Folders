# Story 1.16: Wire exit-criteria and parity completeness gates

Status: done

Created: 2026-05-14

## Story

As a maintainer,
I want governance and completeness gates wired into CI,
so that missing release evidence, unstable idempotency encoding, stale examples, or unsafe tenant cache keys block implementation.

## Acceptance Criteria

1. Given the repository contains C0-C13 exit-criteria evidence artifacts or explicit reference-pending records, when the governance gate runs, then every C0-C13 row maps to an artifact path, owner, status, verification command, and result summary, and missing or malformed evidence fails closed.
2. Given `docs/exit-criteria/` contains approved workshop outputs and placeholders, when the exit-criteria presence gate runs, then it distinguishes approved evidence, bounded `reference_pending` decisions, and invalid placeholders without treating unresolved C3/C4/C6/S-2 policy as silently complete.
3. Given `tests/fixtures/idempotency-encoding-corpus.json` and its schema exist, when the idempotency-encoding equivalence gate runs, then corpus cases validate against the schema and every case is consumed by focused helper or parser-policy tests that preserve null-versus-omitted, ordering, Unicode, percent-decoding, casing, malformed-key, and duplicate-key expectations.
4. Given Story 1.12 owns generated `ComputeIdempotencyHash()` helper behavior, when this story adds gates, then it verifies helper/corpus coverage and deterministic local commands without implementing new SDK generation policy or editing generated client output.
5. Given architecture pattern examples are developer-facing contract, when the pattern-example compilation gate runs, then code snippets or extracted examples in architecture, contract docs, and gate documentation compile or fail with a bounded prerequisite-drift diagnostic.
6. Given cache keys can leak or mix tenants, when the tenant-prefixed cache-key lint runs, then in-process, Dapr state, Redis, distributed-cache, generated-helper, test-fixture, and workflow cache keys present in the repository are rejected unless they are tenant-scoped, explicitly non-tenant, or documented synthetic examples.
7. Given `tests/fixtures/parity-contract.yaml` may be generated by Story 1.13, when the parity completeness gate runs, then every current OpenAPI operation appears exactly once in the parity oracle and every parity row references an existing operation ID, adapter expectation, operation family, idempotency rule, and read-consistency classification.
8. Given Story 1.14 owns Contract Spine drift, generated-client golden-file consistency, parity schema validation, and CI workflow wiring, when this story adds completeness gates, then it extends or reuses that lane without duplicating restore/build/test steps or reimplementing server-vs-spine drift detection.
9. Given Story 1.15 owns safety invariant gates, when this story is complete, then it does not implement sentinel redaction, forbidden-field scanning, output-channel leakage checks, or safety corpus policy outside cache-key and diagnostic metadata needed by its own gates.
10. Given some prerequisites may still be absent, when any gate cannot evaluate because a prior story has not produced its owned artifact or command, then it records exact prerequisite drift and fails closed instead of generating placeholder success, inventing parity rows, or broadening scope.
11. Given CI diagnostics are visible outside the local developer machine, when a governance or completeness gate fails, then logs include only gate names, repository-relative paths, criterion IDs, operation IDs, schema pointers, sample IDs, bounded categories, and safe hashes; logs never echo raw payloads, file contents, diffs, provider tokens, credentials, tenant data, local absolute paths, production URLs, cache key values, or unauthorized-resource hints.
12. Given gates must be locally reproducible, when implementation is complete, then documented repository-root commands run the same exit-criteria, idempotency-encoding, pattern-example, cache-key, and parity-completeness checks without Aspire, Dapr sidecars, Keycloak, Redis, GitHub, Forgejo, provider credentials, tenant seed data, production secrets, network calls, or nested submodule initialization.
13. Given active Story 1.11 through Story 1.15 work may still be dirty or reference-pending, when implementation starts, then the developer inspects current OpenAPI, exit-criteria docs, idempotency fixtures, parity fixtures, contract docs, workflow files, and ready story artifacts before assuming prerequisites are implemented.
14. Given this story wires governance and completeness gates only, when implementation is complete, then it does not add runtime REST handlers, EventStore commands, domain aggregate behavior, provider adapters, Git or filesystem side effects outside deterministic tests/tools, SDK generation policy, parity-oracle derivation, CLI commands, MCP tools, UI pages, Dapr policy conformance, provider live-drift jobs, release publishing, safety invariant scanning, or nested-submodule initialization.

- Unknown C0-C13 rows, missing required C0-C13 rows, duplicate criterion rows, and artifact paths outside approved repository-relative locations fail closed.

### Advanced Elicitation Hardening

- The governance/completeness command must start with a discovery phase that records gate names, canonical inputs, prerequisite ownership, and report paths before validation. Discovery disagreements are `prerequisite_drift`, not implementation choices.
- Evidence-source authority is ordered: architecture C0-C13 definitions, approved `docs/exit-criteria/` records, then story-owned generated fixtures. A lower-authority source cannot silently override a higher-authority criterion, owner, status, or verification command.
- Idempotency coverage must be keyed by stable corpus sample IDs. The gate must fail for unmapped samples, mappings to deleted samples, duplicate mappings, and tests that cover only fixture existence without asserting the equivalence classification.
- Pattern-example compilation is opt-in. The implementation must define marker syntax, documentation globs, excluded generated folders, target framework, project references, and documentation-only classification before scanning broadly.
- Cache-key lint exception records must be reviewed artifacts, not regex shortcuts. Exceptions need owner, reason, scope, rule ID, review status or expiry, evidence link, and a guarantee that diagnostics do not print raw key material.
- Parity completeness must compare structured OpenAPI operation IDs and generated parity rows after schema validation, with distinct failures for missing rows, stale rows, duplicate rows, duplicate operation IDs, and incomplete adapter/read-consistency/idempotency metadata.
- CI annotations, markdown summaries, SARIF, console output, and generated reports must share the same safe diagnostic policy. Absolute local paths, raw cache keys, diffs, file contents, tenant data, provider URLs, credentials, and payload excerpts remain forbidden even in failure artifacts.
