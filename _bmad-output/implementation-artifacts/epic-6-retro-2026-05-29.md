## What Went Well

- **Read-only discipline held.** Every console slice preserved the core MVP boundary: no mutation, no repair, no credential reveal, no file browsing, no raw diffs, no hidden provider payloads, and no unrestricted filesystem views.
- **The wireflow contract paid off.** Story 6.5 created the shared diagnostic-wireflow artifact before the page stories. That reduced UI-only semantics drift in Stories 6.6 through 6.10 and gave review a stable basis for scope checks.
- **Metadata-only and redaction rules became executable.** The epic moved from prose requirements to bUnit sweeps, sentinel leakage checks, disclosure-token assertions, redaction affordance tests, and no-mutation selector gates.
- **The UI test lane grew steadily without losing focus.** The UI lane moved from the early console scaffold through the final 521/521 verification count, with focused component/page tests rather than broad brittle browser checks.
- **Review quality improved across the epic.** Repeated review feedback around File List fidelity, forbidden-touch audits, and status synchronization was absorbed by later stories. Story 6.11 explicitly calls out File List fidelity as a recurring Epic 6 review concern and closes it cleanly.
- **The E2E lane became useful rather than decorative.** The final responsive smoke surfaced a real availability defect: `Workspace` and `FolderDetail` crashed on unreachable read-model transport failures. The fix was minimal, mirrored sibling pages, and was locked into the always-run bUnit lane.
- **Release-evidence discipline was honest.** Story 6.10 and 6.11 both separated automated evidence from release-time human/manual checks with `reference_pending` instead of fabricating pass claims.

## Challenges And Growth Areas

- **Tooling environment friction repeated.** The WSL SDK is not the repo-pinned SDK, so reliable build/test execution had to use the Windows SDK from WSL. Playwright also required Windows Chromium because the WSL distro had no supported Chromium package.
- **Deferred E2E remained outside the default gate.** E2E tests compile and now run locally with the right browser setup, but the lane is still deferred/non-blocking. That was acceptable for Epic 6, but Workstream 7 must decide how to provision and schedule it.
- **Several checks are necessarily release-validation evidence, not CI.** Screen-reader review, forced-colors/high-contrast, color-blindness, zoom, and real-width visual checks still require a release-time human pass. The methods are documented, but evidence is intentionally not claimed yet.
- **The console exposed read-model degradation inconsistency late.** The 6.11 responsive smoke revealed that five pages handled transport failure correctly while two did not. The fix was small, but the pattern suggests future page work should include transport-failure regression tests from the first page story.
- **Story status and commit hygiene required active orchestration.** Child sessions repeatedly stopped at interactive commit prompts; commits had to be selective to avoid unrelated `.claude/scheduled_tasks.lock` and orchestration-state changes.
- **Baseline project reds remained background noise in some earlier story records.** Epic 6 avoided most of the noise by using focused UI lanes, but unresolved cross-epic baseline issues still appear in historical story notes and should be handled by release-readiness gates.

## Key Lessons

1. **Create the UX contract before building page slices.** Story 6.5 reduced ambiguity for the remaining UI stories and should be repeated for any future console expansion.
2. **No-mutation needs both registry and rendered-DOM coverage.** A route can be read-only by intent yet still leak a form, dialog, command marker, or mutation affordance. The final five-selector sweep is the right invariant.
3. **Degraded-state behavior is a first-class correctness requirement.** The operator console must stay useful when projections or APIs are unavailable. Every SDK-backed page should have a transport-failure-to-read-model-unavailable guard and test.
4. **Release-validation evidence must be explicit about gate type.** Accessibility and usability requirements can be legitimate without being a blocking CI gate, but the evidence artifact must name what is automated, what is manual, and what is still pending.
5. **E2E smoke tests should assert structure, not pixels.** The useful browser check was root + single heading + no mutation affordances across routes and widths. It found a real defect without brittle CSS or sleep-based assertions.
6. **Story records are part of the deliverable.** The recurring review issues were documentation mismatches, file-list omissions, and stale counts, not only code defects. Later stories improved because review made those gaps visible.

## Significant Discoveries

- **Responsive structural E2E found a real production availability defect.** `Workspace` and `FolderDetail` did not catch `HttpRequestException` / `TaskCanceledException` and returned HTTP 500 when the read model was unreachable. The fix is already committed in Story 6.11.
- **The hermetic E2E fixture needed a realistic SDK base address.** Without it, SDK-backed pages failed with an unconfigured-base-address exception instead of the transport-failure mode production pages are expected to handle.
- **Accessibility evidence is correctly split.** The automated portion is structural and repeatable; screen-reader, forced-colors, zoom, color-blindness, and real visual responsive checks remain release-time evidence.
- **No fundamental Epic 7 plan change is required.** Epic 7 already owns release-readiness consolidation. The main effect of Epic 6 is to provide concrete artifacts and a few explicit follow-through tasks for that workstream.
