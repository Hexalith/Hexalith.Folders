# Epic 5 Retrospective: Cross-Surface Workflow Parity

## Epic Summary

Epic 5 closed the cross-surface parity loop. The seven stories took a generated REST/SDK contract spine and turned it into four behaviorally-equivalent surfaces — SDK, CLI, MCP, and the existing REST — verified row-by-row through a single shared oracle (`tests/fixtures/parity-contract.yaml`) and a single set of linked test sources. Sprint status shows 7 of 7 Epic 5 stories marked `done`; the retrospective status was `optional` before this review.

The epic delivered:

- **SDK convenience layer** (Story 5.1) — `Hexalith.Folders.Client/Convenience/` with `UploadFileAsync`, `CorrelationAndTaskId`, `FileUpload`, an end-to-end `Hexalith.Folders.Sample`, and `docs/sdk/quickstart.md`.
- **CLI adapter** (Story 5.2) — `Hexalith.Folders.Cli` as a thin System.CommandLine 2.0.8 wrapper over `IClient`: seven command groups, canonical category→exit-code projection, metadata-only output.
- **MCP adapter** (Story 5.3) — `Hexalith.Folders.Mcp` with 47 tools (one per `operation_id`) plus two read-only resources (`folder-tree`, `audit-trail`); kind == canonical category verbatim.
- **Shared oracle consumption** (Story 5.4) — single `tests/shared/Parity/` reader linked into both CLI and MCP test projects; 529 oracle outcome entries iterated per adapter as xUnit theory data.
- **REST/SDK transport parity** (Story 5.5) — transport-parity columns plus the dual-surface `GoldenLifecycleParityTests` running one in-process host against both REST `HttpClient` and the typed SDK `IClient`.
- **Cross-adapter CLI/MCP behavioral parity** (Story 5.6) — single assembly co-references CLI + MCP; cross-adapter symmetry asserted in one iteration of the loop, transitive Story 5.4 claim collapsed into a single assertion site.
- **Mixed-surface handoff** (Story 5.7) — one logical task lifecycle split across all four surfaces against one in-process host; identity (`task_id` / `correlation_id` / `idempotency_key`) preserved end-to-end.

## What Went Well

- **Thin-adapter discipline held across every story.** Each adapter csproj references only `Hexalith.Folders.Client`; every command, tool, or convenience reduces to one `IClient` call. Reviewer verified row-by-row that no surface invents fields, states, or operations absent from the Contract Spine.
- **Single source of truth, single set of tests.** The Story 5.4 shared reader under `tests/shared/Parity/` became the spine of the epic. Stories 5.5/5.6/5.7 extended it additively (transport columns, golden-lifecycle steps, mixed-surface scenarios) without forking the loader, copying the fixture, or editing `Hexalith.Folders.slnx`.
- **Oracle-driven coverage scaled.** 47 operation rows × 43 canonical categories × 529 outcome-mapping entries are iterated as xUnit theory data. Drift-injection sanity (intentionally flipping one cell and confirming the suite fails the expected count of cases) was repeated in every late-epic story.
- **Hermeticity preserved throughout.** No live Dapr, Keycloak, Redis, GitHub/Forgejo, providers, or network in any test. In-process host on `127.0.0.1:0` with in-memory repository/gateway/read-models stood up the dual-surface and mixed-surface runs. No recursive submodule init; sibling gitlink drift was left alone every commit.
- **Review cycles converged fast.** Most stories needed one auto-fix review pass; Story 5.5 needed a second pass that closed three clearly-testable AC items (`audit_metadata_keys` assertion, `context_returned` REST terminal class, Newtonsoft→EnumMember reflection). Reviewer-applied auto-fixes (camelCase wire shape, resource hygiene for leaked `HttpClient`, stale test-count corrections) became a reliable safety net.
- **Negative scope was preserved as a deliverable.** Every story explicitly recorded out-of-scope items (e.g. "the `fileOperationKind` integer-serialization issue is Contracts/Epic-1 follow-up"; "audit-family REST routes are deferred to Epic 6 Story 6.1"). The known-drift artifacts (`KnownRestSurfaceGap`, `EndpointNameAliases`, `RestInspectionOperationId` substitution) surface gaps in tests rather than papering over them.
- **Lessons from the Epic 1 retro landed.** Executable boundaries over prose, metadata-only everywhere, explicit negative scope, sprint metadata kept current — all visible in every Epic 5 story record.

## Challenges And Growth Areas

- **REST surface gap of 19 operations is now a known load-bearing constraint.** Story 5.5 documented it: only 28 of 47 `operation_id` rows have implemented `/api/v1` routes. The 19 missing rows include the **entire audit family** (`ListAuditTrail`, `GetAuditRecord`, `ListOperationTimeline`, `GetOperationTimelineEntry`), ACL list/update (`ListFolderAclEntries`, `UpdateFolderAclEntry`), several queries (`GetProviderBinding`, `GetRepositoryBinding`, `GetWorkspaceRetryEligibility`, `GetWorkspaceTransitionEvidence`, `GetProjectionFreshness`), mutators (`CreateFolder`, `ConfigureProviderBinding`), and six diagnostics endpoints. Epic 6 Story 6.1 owns the audit family; the rest still need owning stories.
- **Carry-over reds across the epic.** Three pre-existing test failures appeared in every story's Dev Notes:
  1. `ClientGenerationTests.GeneratedClientAndHelpersMatchIsolatedRegeneration` — a 6-line whitespace-only diff between the committed `Generated/HexalithFoldersClient.g.cs` and a fresh NSwag regeneration. Caused by `.editorconfig` post-generation normalization (trim trailing whitespace + add final newline) that the test's regen pipeline does not apply. Not fixable without editing `Generated/` (forbidden).
  2. `BranchRefPolicyEndpointTests.GetBranchRefPolicyShouldUseSafeDenialEnvelopeForTenantMismatch` — latent at Story 5.4 baseline (`75e9782`); neither test nor production source changed since. Unrelated to Epic 5.
  3. `Hexalith.Folders.Testing.Tests.ProjectReferencesFollowAllowedDependencyDirection` — surfaced after Story 5.5 added `Hexalith.Folders.Client` as a `ProjectReference` from IntegrationTests; the dependency-direction policy in `Hexalith.Folders.Testing.Tests` was not updated to match. Out of scope for every Epic 5 story.
- **Cross-adapter casing observation (Story 5.6).** The CLI's `ResultRenderer.RenderProblem` emits the error category via `Category.ToString()` → `Folder_acl_denied` (PascalCase) while MCP emits via `[EnumMember]` → `folder_acl_denied` (canonical snake_case). The canonical snake_case still surfaces on the CLI via the server-supplied `problem.Code` echo, so AC verbiage held — but a strict "CLI category field equals the EnumMember snake_case" assertion would catch this divergence. Out of scope for Story 5.6; flagged for a CLI source fix.
- **Authoritative-set trap was a live hazard for every story.** Architecture.md §"Adapter Parity Contract" carries an abbreviated 13-row failure-kind summary that misspells `unknown_provider_outcome` as `provider_outcome_unknown`. Every story Dev Notes block warns against asserting from architecture.md; the oracle YAML is the only source of truth. Worth repeating in Epic 6's Dev Notes templates.
- **Story 5.1's authoritative-spine-vs-prose divergence.** The story's prose described `transportOperation` as a polymorphic body union; the generated `FileMutationRequest` actually models it as a string discriminator with content carried via `[JsonExtensionData] AdditionalProperties`. The convenience helpers had to follow the implemented spine, not the prose. Story 5.1 also flagged that the generated `fileOperationKind` enum lacks a per-property `StringEnumConverter` and serializes as integer rather than `add`/`change` — deferred to Contracts/Epic-1.
- **Architecture.md D-9 file-transport prose has drifted from the implementation.** Story 5.1 noted that D-9 names `PutFileInline`/`PutFileStream`/`UploadFileAsync(stream)` and multipart shapes that don't exist in the actual spine. The concept (inline small / stream large, single-call DX) holds; the surface description is stale.
- **In-process host stubs limit which negative paths can be provoked.** Stories 5.5 and 5.7 found that the in-process gateway flattens the aggregate's `IsRejection` flag, so a real `idempotency_conflict` doesn't surface as REST 409 / SDK exception in the hermetic fixture. Same with `folder_acl_denied` — the layered-authorization stack short-circuits at `AllowingEventStoreAuthorizationValidator`. Story 5.7 surfaced this in Dev Notes and asserted the aggregate-side invariant (`EventsAppended` + `TryGetIdempotencyFingerprint`) instead.
- **Lifecycle snapshot task-binding required a workaround in Story 5.7.** `InMemoryFolderRepository.SaveLifecycleSnapshot` auto-binds the snapshot's `EvidenceScope.TaskId` to the last mutator's `task_id`, but query operations don't accept a `task_id` parameter, so the handler's `task_mismatch` check blocked every downstream query. Story 5.7 re-seeds the snapshot post-chain with `TaskId=null` to simulate the production projection's metadata-only snapshot. Not a regression — but a real test-fixture wrinkle.
- **WSL .NET SDK build environment.** Every story noted the WSL-native `.NET` SDK is 10.0.108 and fails the `global.json` `10.0.300` pin. Build/test had to go through the Windows SDK (`/mnt/c/Program Files/dotnet/dotnet.exe`) from WSL. The memory record `dotnet-windows-sdk-wsl.md` captures this.
- **`ImplementedRestOperationCount = 28` is a magic constant.** Story 5.5's review flagged it: the bookkeeping for `KnownRestSurfaceGap` (19 items) + `EndpointNameAliases` (3 items) + the count must stay in lock-step. The both-direction invariant test already enforces it; the constant adds maintenance without adding safety.

## Key Lessons

1. **A single oracle plus linked test sources beats per-surface re-encoding.** The shared reader under `tests/shared/Parity/` linked into four consuming test projects (CLI, MCP, Client, Server, IntegrationTests) collapsed N×M test-design decisions into one. Stories 5.4 → 5.5 → 5.6 → 5.7 each extended it additively without forking; that compounding pays.
2. **Drift-bite sanity becomes the test you can trust.** Every late-epic story manually flipped one cell, confirmed the suite failed the expected count of cases, reverted, and recorded the count. That practice catches both "the test is tautological" and "the assertion is too weak."
3. **The Contract Spine wins over prose every time.** When the implemented spine and architecture.md / story prose diverge, follow the spine and record the prose drift as a documentation TODO. Stories 5.1 (transport shape, `fileOperationKind` enum converter), 5.5/5.6 (architecture's abridged failure-kind list misspelling `unknown_provider_outcome`), and 5.5 (architecture D-9 file-transport naming) all hit this.
4. **Surface fixture limitations rather than papering over them.** `KnownRestSurfaceGap`, `RestInspectionOperationId`, the in-process gateway response-flattening note, the layered-authorization short-circuit note — every one is documented in test class remarks plus Dev Notes. When the gap closes, the assertions extend; no test rewrite needed.
5. **CLI command names are not kebab-case of `operation_id`; MCP tool names are.** This is the single most common cross-adapter mistake and it bit Stories 5.4, 5.6, and 5.7 at the design stage. Future cross-adapter tests need a small explicit `operation_id → (CLI argv, MCP tool method)` map inline; do not encode it as an oracle column (contract-workflow concern, not a test concern).
6. **Hermetic CLI testing wants `CliDependencies` composition, not the `CliTestHarness.UseRealClient` canned-response path.** Story 5.7 settled this: in-process-host-bound `IClient` factory + `TestCliConsole` + linked credential resolver composes the CLI's real pipeline against the in-process host. The harness's canned-response path remains the right tool for per-adapter pre-SDK behavior tests.
7. **`[EnumMember]` reflection beats `Enum.Parse`.** The snake_case oracle string → `CanonicalErrorCategory` resolution via `EnumMemberAttribute.Value` is the canonical pattern. Story 5.5 added `ResolveCanonicalCategoryWireValue` to replace an incidental Newtonsoft.Json `StringEnumConverter` serialization path; Story 5.6 mirrored it. Don't reach for Newtonsoft for one string conversion.

## Significant Discoveries

- **The REST surface is 28/47 implemented.** `KnownRestSurfaceGap` (19 items) is the canonical drift record. No prior epic surfaced this aggregate count; Story 5.5 made it visible. Epic 6 Story 6.1 owns the audit-family subset (4 ops); the remaining 15 (ACL, queries, mutators, diagnostics) still need owning stories or release-readiness consideration.
- **The cross-adapter casing divergence is real.** CLI emits PascalCase via `Category.ToString()`; MCP emits canonical snake_case via `[EnumMember]`. The AC text passed because canonical snake_case still appears on CLI via the server-supplied `problem.Code` echo, but a strict assertion would catch the divergence.
- **In-process host idempotency-conflict propagation is fixture-limited.** The aggregate's `IsRejection` flag is flattened by `InProcessEventStoreGatewayClient`. Production gateway behavior is expected to translate the rejection; the hermetic fixture cannot. Story 5.7 surfaced this and asserted at the aggregate ledger layer — a correct fallback per the story's negative-scope directive.
- **The Epic 1 retro called out that "adapter parity has two layers"; Epic 5 made the second layer (CLI/MCP behavioral) executable.** Now both layers ride on the same oracle. The remaining question for Workstream 7 release-readiness is: does the parity oracle's row coverage need to match the implemented REST surface, or does the oracle deliberately lead the implementation? Current pattern: oracle leads, implementation catches up; drift is visible by design.
