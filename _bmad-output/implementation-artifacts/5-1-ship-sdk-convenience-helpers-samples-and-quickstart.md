# Story 5.1: Ship SDK convenience helpers, samples, and quickstart

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As an SDK consumer,
I want ergonomic helpers, samples, and quickstart material,
so that I can use the canonical lifecycle without learning internal transport details.

## Acceptance Criteria

Source epic AC (epics.md#Story-5.1):

> **Given** generated SDK methods exist
> **When** helpers and samples are added
> **Then** upload convenience, idempotency guidance, correlation/task ID handling, and a local AppHost sample are documented
> **And** helpers do not introduce lifecycle semantics absent from the Contract Spine.

Decomposed, testable acceptance criteria:

1. **Upload convenience helper** — A hand-written `UploadFileAsync`-style convenience over the generated `AddFileAsync`/`ChangeFileAsync` methods selects the inline vs streamed `transportOperation` shape automatically based on content length (inline ≤ 262144 bytes; streamed above), so callers pass a `Stream`/`ReadOnlyMemory<byte>` plus metadata and never hand-build the polymorphic `FileMutationRequest.transportOperation` union. The helper lives outside `Generated/` and adds **no** new request fields, states, operations, or behaviors not already present in the Contract Spine.

2. **Idempotency guidance** — SDK consumers are given a documented, supported path to compute and supply idempotency keys using the existing generated `ComputeIdempotencyHash()` helpers (returns `sha256:<hex>`); convenience does **not** auto-generate or silently substitute idempotency keys. Replay (`idempotentReplay: true`) and `idempotency_conflict` semantics are explained, not re-implemented.

3. **Correlation / Task ID handling** — Convenience for correlation-ID and task-ID sourcing follows the Adapter Parity Contract: correlation ID is caller-provided OR resolved from a registered provider OR, only when neither is set, a fresh ULID generated by the SDK; task ID is caller-provided and is **never** SDK-generated (required for task-scoped operations: prepare, lock, file mutation, commit, release). Sourcing precedence is documented and unit-tested.

4. **Local AppHost sample** — `samples/Hexalith.Folders.Sample` is upgraded from the 4-line scaffold into a runnable end-to-end demonstration that drives the canonical lifecycle through `IClient` + the convenience helpers, pointed at the local `Hexalith.Folders.AppHost` Aspire topology. The canonical sequence (per the Epic 5 golden flow) is: configure provider binding → validate provider readiness → create a repository-backed folder (or bind an existing repository) → prepare workspace → lock → add/change file via the upload helper → commit → query context/status → inspect audit. (Workspace prepare requires a *ready repository-backed* folder — do not call prepare on a plain `CreateFolder` result.) The sample's documentation explains how to launch the AppHost and run the sample.

5. **Quickstart documentation** — A new SDK quickstart document covers: DI registration (`AddFoldersClient`), base-address/auth wiring, the upload convenience helper, idempotency-key guidance, correlation/task-ID sourcing, and the local-AppHost run instructions, with copy-pasteable, **metadata-only** code examples (no secrets, tokens, raw file content, diffs, or local absolute paths).

6. **Parity-preserving guardrail (regression gate)** — Tests assert the helpers are thin: every lifecycle effect the helpers trigger maps 1:1 to an existing generated operation; no helper introduces a request field, state, terminal outcome, error category, or operation absent from `hexalith.folders.v1.yaml`.

7. **Hermetic build/test** — The sample compiles and its tests run with no provider credentials, no running Dapr sidecars, no Keycloak/Redis, no network calls, and no nested submodule initialization. Live end-to-end execution against the AppHost is documented as an opt-in manual/integration path, not a blocking unit gate.

### Critical guardrail — no lifecycle semantics beyond the spine

The defining constraint of this story (epic AC): **helpers do not introduce lifecycle semantics absent from the Contract Spine.** Every convenience must reduce to a call to an already-generated operation with arguments that already exist in `src/Hexalith.Folders.Contracts/openapi/hexalith.folders.v1.yaml`. Do not invent new states, fields, error categories, retry semantics, or operations. If a desired convenience would require a spine change, it belongs in a different story (Contracts/Epic 1 territory), not here.

### Current state — what already exists (do not reinvent)

- **Typed SDK is generated and registered.** `src/Hexalith.Folders.Client/Generated/HexalithFoldersClient.g.cs` exposes the `IClient` interface + `Client` implementation (namespace `Hexalith.Folders.Client.Generated`). Async-only, `CancellationToken` overloads, injected `HttpClient`, `useBaseUrl: false`. DI entry points already exist: `FoldersClientServiceCollectionExtensions.AddFoldersClient(...)` and `FoldersClientOptions` (`BaseAddress`, section `"Folders"`). Auth is intentionally out of scope of the module — callers attach a bearer-token `DelegatingHandler` on the returned `IHttpClientBuilder`.
- **Generated mutation signatures** all take the header triple as explicit string parameters, e.g.:
  - `AddFileAsync(string folderId, string workspaceId, string idempotency_Key, string x_Correlation_Id, string x_Hexalith_Task_Id, FileMutationRequest body, CancellationToken)` (same shape for `ChangeFileAsync`, `RemoveFileAsync`).
  - `CreateFolderAsync(idempotency_Key, x_Correlation_Id, x_Hexalith_Task_Id, CreateFolderRequest, …)`, `PrepareWorkspaceAsync`/`LockWorkspaceAsync`/`ReleaseWorkspaceLockAsync(folderId, workspaceId, …)`, `CommitWorkspaceAsync(...) → CommitWorkspaceAccepted`.
  - The SDK does **not** currently auto-source any of the three headers — callers pass them explicitly. Tightening that into a documented sourcing model is this story's job.
- **`AcceptedCommand` response** carries `acceptedAt`, `correlationId`, `taskId`, `status`, and `idempotentReplay` (bool). Use these real fields in samples/tests; do not invent response shapes.
- **File transport is one operation with a polymorphic body**, not separate inline/stream endpoints and not multipart. `FileMutationRequest` has `transportOperation` (the inline-vs-streamed union), `fileOperationKind` (Add/Change/Remove), and `byteLength`. The two transport shapes carry: inline → `contentBytes` + `contentMediaType`; streamed → `observedLength` + `stagingReference` + `observedContentHashReference` + `mediaType`. The 256 KB (262144-byte) inline boundary and the REST `413` + `X-Hexalith-Retry-Transport: stream` retry contract are real and were finalized in Story 4.6. (The request-side `X-Hexalith-Retry-As` header is a separate retry-allocation hint, not the transport-substitution header.) (Note: architecture.md's D-9 names `PutFileInline`/`PutFileStream`/`UploadFileAsync(stream)` and multipart — that text **drifted**; the implemented spine is authoritative. Keep the *concept* — inline small / stream large, single-call DX — but match the actual `FileMutationRequest` shape.)
- **Idempotency helpers are generated.** `Generated/HexalithFoldersIdempotencyHelpers.g.cs` adds `ComputeIdempotencyHash()` to each mutating request DTO; the canonical hasher is hand-written at `src/Hexalith.Folders.Client/Idempotency/HexalithIdempotencyHasher.cs`. Output format is `sha256:<lowercase-hex>`. **Consumers must use these, never re-implement canonicalization.** `HexalithFoldersGeneratedArtifacts.HelperSchemaVersion` is a 16-hex fingerprint for detecting helper-signature drift across SDK versions.
- **Sample today is a 4-line scaffold** (`samples/Hexalith.Folders.Sample/Program.cs` prints the module name) with a compile-only smoke test (`SampleSmokeTests.SampleProjectCompilesWithoutExternalServices`). Both projects + the Client + Client.Tests are already wired into `Hexalith.Folders.slnx` under `/samples/`. `Hexalith.Folders.Sample.csproj` is `OutputType=Exe`, `IsPackable=false`, references the Client.
- **AppHost / Aspire** live at `src/Hexalith.Folders.AppHost` (orchestrates Server + Workers + UI + EventStore + Tenants + Keycloak + Dapr + Redis) and `src/Hexalith.Folders.Aspire`. Launch with `dotnet run --project src/Hexalith.Folders.AppHost`; dashboard at `https://localhost:17000`.

### Adapter Parity Contract — sourcing rules the helpers MUST honor

[Source: architecture.md#Adapter-Parity-Contract, decisions A-2/A-9/A-10]

- **Idempotency key (SDK):** caller-provided via method parameter, OR resolved from a registered key-source abstraction; **NEVER auto-generated by the SDK.** Same key + equivalent payload ⇒ same logical result (`idempotentReplay: true`); same key + different payload ⇒ `idempotency_conflict`.
- **Correlation ID (SDK):** caller-provided via `X-Correlation-Id`, OR resolved from a registered provider; when neither is set, the SDK generates a fresh ULID per call. This is the one value the SDK may synthesize.
- **Task ID (SDK):** caller-provided via `X-Hexalith-Task-Id`; the SDK does **not** generate it; it is required for task-scoped operations (prepare, lock, file mutation, commit, release) and must not be downgraded to empty.
- Cross-adapter invariant: for the same input + authoritative tenant + same correlation/task/idempotency triple, the (canonical category, code, retryable, clientAction) returned by SDK/REST/CLI/MCP are identical. The helpers must not perturb this.

### Previous-story intelligence (lessons to carry in)

- **Story 1.12 (NSwag SDK generation):** generated files are off-limits for hand edits; idempotency helpers exist only for mutating ops that declare `x-hexalith-idempotency-equivalence` — never add helpers to queries. Helper parameter order follows spine path-params then the equivalence list (e.g., `PrepareWorkspaceRequest.ComputeIdempotencyHash(folderId, workspaceId, taskId)`). `docs/contract/sdk-generation-and-idempotency-helpers.md` line 83 explicitly defers "any future `UploadFileAsync(stream)` convenience API" to Epic 5 — this story.
- **Story 4.6 (file transport):** streamed requests must carry observed evidence (`observedLength`, `stagingReference`, `observedContentHashReference`) — descriptor-only was rejected in review. `byteLength` is the decoded byte count, not a base64 string length. Inline-over-boundary returns a safe `413` retry hint without disclosing the byte limit, path, or content. Never put file bytes/base64/diffs in events, logs, or examples.
- **Story 4.11 (correlation/task/idempotency propagation):** the header triple is validated before body parsing on REST; task ID must fail closed (not be coerced to empty) when missing on task-scoped ops; replay no-op payloads must survive the wire so replay evidence (`idempotentReplay`) is observable.
- **Story 4.12 (commit):** commit only from `changes_staged`; unknown provider outcomes become `unknown_provider_outcome`/`reconciliation_required` — surface these states truthfully in the sample, never paper over them.
- **Epic 1 retro:** prefer executable boundaries over prose; record negative scope explicitly (which future story owns deferred bits); metadata-only is mandatory everywhere including docs examples and test output.
