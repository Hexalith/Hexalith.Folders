# Story 5.7: Validate mixed-surface handoff scenario

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As an automation developer,
I want one task lifecycle to move between REST, SDK, CLI, and MCP using the same IDs,
so that real integrations can hand off work without losing state or auditability.

## Acceptance Criteria

Source epic AC (epics.md#Story-5.7):

> **Given** all four surfaces are available
> **When** provider readiness, create/bind, prepare, lock, write, query, commit, status, and release are split across surfaces
> **Then** task ID, correlation ID, operation IDs, audit records, and terminal state remain coherent
> **And** any surface-specific drift in idempotency replay or error category fails the scenario.

Decomposed, testable acceptance criteria:

1. **Single in-process host, four surface clients, one logical task.** A single `Hexalith.Folders.IntegrationTests/MixedSurfaceHandoff/` test class boots one in-process server host (reuse the Story 5.5 `TestHost` / `StartHostAsync` pattern from `tests/Hexalith.Folders.IntegrationTests/EndToEnd/GoldenLifecycleParityTests.cs` — `WebApplication.CreateSlimBuilder` on `127.0.0.1:0`, in-memory repository/gateway/lifecycle/tenant/permissions stubs, `MutableTenantAndClaimContext`) and exposes **four** clients targeting that same host URI: (a) raw `HttpClient` (REST), (b) generated `Hexalith.Folders.Client.Generated.IClient` (SDK) constructed `new Client(new HttpClient { BaseAddress = hostUri })`, (c) CLI dependencies composed with an `IClient` constructed the same way against `hostUri` (no edits to `src/Hexalith.Folders.Cli/**` — compose `CliDependencies` directly from IntegrationTests via the `InternalsVisibleTo` grant Story 5.6 added; the Story 5.6 linked-compile CLI harness sources stay available unchanged), (d) MCP `ToolPipeline` built via `TestSupport.Pipeline(new GeneratedClient(new HttpClient { BaseAddress = hostUri }))` (the linked-compile MCP test support from Story 5.6). All four clients hit the **same** `hostUri`, so every surface step round-trips through the same `/process` endpoint and the same in-memory aggregate/read-models.

2. **One shared identity triple (`task_id`, `correlation_id`, `idempotency_key`) flows across surfaces unchanged.** A single test-defined `MixedSurfaceIdentity` record fixes the three identifiers for the whole scenario: `task_id = "task-handoff-{ulid}"`, `correlation_id = "corr-handoff-{ulid}"`, plus per-step `idempotency_key = "key-{step}-{ulid}"` values (one per mutating step). For every surface step, the test asserts: (a) the wire request carries `X-Hexalith-Task-Id == task_id` (REST raw header; SDK/CLI/MCP propagated via the SDK method parameter); (b) the wire request carries `X-Correlation-Id == correlation_id` (REST raw header; SDK/CLI/MCP propagated via the SDK method parameter); (c) the response **echoes** `X-Correlation-Id == correlation_id` **unchanged** on every surface; (d) for the mutating surfaces (REST/SDK/CLI/MCP), `X-Hexalith-Task-Id == task_id` is echoed unchanged on the response header (REST) or surfaced verbatim on `AcceptedCommand.TaskId` (SDK) / CLI stderr `correlation-id`/`task-id` lines / MCP result envelope. **No surface generates a new task_id or correlation_id mid-scenario** — caller-supplied identity is preserved end-to-end (caller→SDK→wire→aggregate→response). The wire-observed values are read from the same `InProcessEventStoreGatewayClient.ProcessCalls` counter so the test proves the aggregate received the caller's identity exactly.

3. **Cross-surface operation-identity chain: each step is pinned to its oracle `operation_id`.** The handoff scenario is a single ordered list of `(StepName, OperationId, ExecutingSurface)` pairs covering the canonical mutating chain split across four surfaces: e.g. `provider_readiness` via **REST**, `repository_binding` (CreateRepositoryBackedFolder) via **SDK**, `archive` (ArchiveFolder) via **CLI**, `lifecycle_inspection` (GetFolderLifecycleStatus) via **MCP** (and additional steps as the REST surface gap permits — see Dev Notes "Implemented operation envelope"). Each step asserts: (a) the `operation_id` is a row in `tests/fixtures/parity-contract.yaml` via `ParityScenarios.Row(operationId)`; (b) `adapter_expectations` contains the executing surface (`rest` / `sdk` / `cli` / `mcp`); (c) the SDK method name resolves to `{operation_id}Async` on `IClient` (operation identity, mirror Story 5.5 AC #2); (d) the CLI command argv invoked maps to that `operation_id` per the Story 5.6 explicit `operation_id → (CLI argv, MCP tool reference)` table (e.g. `ArchiveFolder` ⇒ `folder archive`); (e) the MCP tool method on `Hexalith.Folders.Mcp.Tools.*` (e.g. `FolderTools.ArchiveFolder`) wraps the SDK call to the matching `{operation_id}Async`. **Operation identity is the same across all four surfaces for the same logical step.**

4. **Cross-surface state coherence: REST step's write is visible to a downstream SDK/CLI/MCP read.** Drive a mutating step on surface **A** (e.g. REST → `ArchiveFolder`), then drive a query step on surface **B** (e.g. SDK → `GetFolderLifecycleStatus`) and surface **C** (e.g. MCP → `GetFolderLifecycleStatus`) and surface **D** (e.g. CLI → `folder status`). All three query surfaces observe the **same** lifecycle state (the post-archive state, per the in-process lifecycle read model) and the **same** `correlationId` recorded against the evidence snapshot (`EvidenceScope.CorrelationId`). Inversely, drive a mutating step on **MCP**/**CLI**/**SDK** and verify the REST query sees it. **Surface choice for the write does not change what downstream surfaces read** — exactly the FR51 invariant ("cross-surface equivalence for ... status transitions, audit records, operation IDs").

5. **Cross-surface idempotency-replay parity: same key, different surface, same logical result.** For a representative mutating step (e.g. `ArchiveFolder`), execute it through surface **A** with `(task_id, correlation_id, idempotency_key) = T`. Then **replay** the same logical command with the **same** `idempotency_key` (and same task/correlation) through surface **B**. Assertions: (a) surface **B**'s reponse marks the call as an **idempotent replay** (`idempotentReplay == true` in the response envelope where the server populates it, or — where the in-process projection's snapshot already reflects the operation — surface B observes the post-A state with no second write-side event; the `InProcessEventStoreGatewayClient.ProcessCalls` counter must equal **A's-process-call-count + 0 or 1** depending on the server's replay handling, **never two distinct events for the same `idempotency_key`**); (b) the canonical category surfaced is `success` (or `idempotency_replay` per the server's `idempotentReplay` marker on `FolderAuditEndpointFilter.IsIdempotentReplay`), **not** `idempotency_conflict`; (c) for a **conflicting payload replay** (same `idempotency_key`, different request payload) on a second surface, all four surfaces produce the canonical `idempotency_conflict` category (CLI exit `68`, MCP failure kind `idempotency_conflict`, SDK throws `HexalithFoldersApiException<ProblemDetails>` with category `idempotency_conflict`, REST returns `409` with canonical RFC 9457). **Cross-surface idempotency replay is a strictly aggregate-side invariant; no surface gets a free pass to silently de-duplicate locally.**

6. **Cross-surface error-category parity: same provoked failure, four surfaces, identical canonical category.** For each of three provoked failures, drive the **same** condition through all four surfaces and assert the surfaced canonical category is **byte-for-byte identical**:
   - **`authentication_failure`** — withhold the tenant claim (`TestHost.StartAsync(tenantId: null, principalId: null)` as Story 5.5 used). REST → `401` RFC 9457 with `category == "authentication_failure"`; SDK → `HexalithFoldersApiException<ProblemDetails>` with `problem.Category` mapping to the same canonical (via `[EnumMember]` value resolved like Story 5.5's `ResolveCanonicalCategoryWireValue`); CLI → stderr contains `authentication_failure` AND exit code `65` (oracle outcome row for `ArchiveFolder` / `authentication_failure`); MCP → result envelope `kind == "authentication_failure"` AND `code == "authentication_failure"`.
   - **`folder_acl_denied`** — seed tenant but not folder ACL (withhold `SeedPermissions`). All four surfaces emit canonical `folder_acl_denied` (CLI exit `66`, MCP kind `folder_acl_denied`, SDK + REST canonical `category == "folder_acl_denied"`). Reuse the Story 5.6 cross-adapter post-SDK pattern (`CrossAdapterBehavioralParityTests` post-SDK categories) extended to REST + SDK in the same iteration.
   - **`idempotency_conflict`** — drive the same `idempotency_key` with conflicting payloads through two different surfaces (per AC #5). All four surfaces emit canonical `idempotency_conflict` (CLI exit `68`, MCP kind `idempotency_conflict`).

   Surface drift on **any** of these categories on **any** of the four surfaces fails the scenario.

7. **Cross-surface audit/timeline coherence: the lifecycle-status surrogate carries the cumulative state.** The audit-family endpoints (`ListAuditTrail`/`GetAuditRecord`/`ListOperationTimeline`/`GetOperationTimelineEntry`) still have no `/api/v1` route (Story 5.5 REST surface gap; documented in `tests/Hexalith.Folders.IntegrationTests/EndToEnd/GoldenLifecycleParityTests.cs` `ProjectionSurfacedAuditKeys` boundary). **Per the same drift-aware reconciliation Story 5.5 used,** the mixed-surface handoff scenario uses `GetFolderLifecycleStatus` as the cross-surface audit-inspection surrogate. After all mutating steps complete (split across REST/SDK/CLI/MCP), query the lifecycle status **once per surface** and assert: (a) the returned `lifecycle_state` reflects the cumulative chain of mutations (e.g. post-archive state); (b) the surrogate's `EvidenceScope.CorrelationId` matches the **last** mutating step's `correlation_id`; (c) the response carries **none** of the forbidden content patterns from `GoldenLifecycleParityTests.ForbiddenContentPatterns` (no secrets/tokens/raw file contents/diffs/provider payloads/absolute paths) — metadata-only invariant across surfaces. When the audit-family endpoints land in a future story, the surrogate's `RestInspectionOperationId` substitution flips to `ListAuditTrail` with no test code change (mirror Story 5.5's `GoldenLifecycleStep.RestInspectionOperationId` pattern). Record this boundary in the new test class doc-comments verbatim.

8. **Cross-surface terminal-state coherence: the chain's terminal-state class matches the oracle for each step's surface.** Per Story 5.5 AC #6, each operation's HTTP outcome matches its `transport_parity.terminal_states` class: mutating → `accepted` (`202`), query_status → `projected` (`200`). For the mixed-surface chain, every step on every surface lands at its oracle-prescribed terminal-state class:
   - mutating step on REST → `202 Accepted` + canonical `AcceptedCommand` body;
   - mutating step on SDK → `AcceptedCommand` returned (no exception);
   - mutating step on CLI → exit code `0` + stderr `correlation-id:` line;
   - mutating step on MCP → result envelope with `correlationId` populated, no `kind` field (success envelope);
   - query step on any surface → `200 OK` (or SDK return value / CLI stdout JSON object / MCP success envelope).

   The chain's **final** terminal state (after all mutating steps) is observable on all four query surfaces as the same `lifecycle_state` (AC #4).

9. **Hermetic, oracle-driven, test-only — zero edits to production code or shared fixtures.** Build and run with **no** provider credentials, **no** Dapr/Keycloak/Redis sidecars, **no** GitHub/Forgejo network, **no** nested submodule init. The only edits permitted are inside `tests/Hexalith.Folders.IntegrationTests/` (new test file under `MixedSurfaceHandoff/`, possibly a small new shared helper under `tests/shared/Parity/` if a `(operation_id → (CLI argv, MCP tool method))` mapping helper proves cleaner than inline lookup — link into IntegrationTests at the same time). No edits to:

   - Existing test sources under `tests/Hexalith.Folders.Cli.Tests/`, `tests/Hexalith.Folders.Mcp.Tests/`, `tests/Hexalith.Folders.Client.Tests/`, `tests/Hexalith.Folders.Server.Tests/`. The Story 5.6 linked CLI harness + linked MCP TestSupport are **read-only** from this story's perspective (they may not be reshaped); we compose `CliDependencies` directly in IntegrationTests with a real `IClient` over the in-process `hostUri` rather than reworking the harness.
   - `Hexalith.Folders.slnx` — no new project (mirror Story 5.5/5.6's linked-compile + project-reference approach); no inline package `Version` attributes (central `Directory.Packages.props`); no recursive submodule init commands.

   If a test reveals a real cross-surface state-coherence drift, surface it in **Dev Notes** — do **not** "fix" it by editing the oracle, the server, a client, or a generated artifact.

10. **Completeness + drift guards — "any surface drift fails loudly."** Mirror the Story 5.5 / 5.6 row-level guards on the mixed-surface side:
    - **Per-step oracle pin (AC #3):** each handoff step's `operation_id` must exist in `ParityOracle.Rows`; an unknown `operation_id` in the scenario list fails on first access (mirror Story 5.5's `GoldenLifecycle.BuildAndValidate`).
    - **Surface coverage (AC #1, #3):** every step's `ExecutingSurface` is in `{ "rest", "sdk", "cli", "mcp" }` AND the operation's `adapter_expectations` contains that surface; a step bound to a surface the oracle does not expect there fails.
    - **Cross-surface category-set membership (AC #6):** the surfaced canonical category on each surface ∈ that operation's `transport_parity.error_code_set` (oracle row); an out-of-set category fails.
    - **Cross-surface identity invariants (AC #2):** task_id observed on the wire equals task_id supplied by the caller; correlation_id observed in the response equals correlation_id supplied by the caller; non-equality on any surface fails. Surface-locally-generated identifiers (e.g. SDK fresh ULID when caller omits correlation; CLI fresh ULID per invocation) are **forbidden inside the handoff scenario** because the scenario supplies all three identifiers explicitly — the test asserts the supplied identifiers, not the regenerate path (which is covered by Stories 5.5/5.6).
    - **Drift-bite sanity:** during local Task 5 verification, temporarily flip one expected value (e.g. swap correlation_id mid-scenario, or change the executing-surface tag on one step, or inject a payload conflict expecting `idempotency_conflict` but assert success) and confirm the scenario fails on the flipped step with a row-level error message naming the step + surface. ### Cross-surface idempotency replay: what to assert

PRD §FR41/FR42 + project-context: "A repeated call with the same idempotency key and equivalent payload must return the same logical result; the same key with a conflicting payload must return an idempotency conflict." `src/Hexalith.Folders.Server/FolderAuditEndpointFilter.cs` declares `IsIdempotentReplay` on the audit projection side. The exact server-side reach is in scope **only for the assertions** — do **not** edit the server to make a replay test pass.

For the cross-surface idempotency-replay AC #5:

- **Same key + equivalent payload across surfaces:** the second-and-later surface invocations either (i) hit the server's replay path and the response is marked `idempotentReplay == true` in the audit envelope; (ii) bypass the aggregate (the in-memory repository de-duplicates by event id / idempotency key); or (iii) reach the aggregate but the aggregate's idempotency guard short-circuits to the prior result. The test should assert: (1) `host.Gateway.ProcessCalls` does **not** increment by 4 (one per surface) — that would indicate every surface produced a fresh event. (2) None of the four surfaces reports `idempotency_conflict`. (3) All four surfaces report `success` or `idempotent_replay` (the canonical categories the server's audit projection emits for replay). The exact ProcessCalls count is server-implementation-defined; **don't** hard-code "must equal 1" — assert the **conflict-free + no-extra-events** invariant.
- **Same key + conflicting payload across surfaces:** every surface must surface canonical `idempotency_conflict` (oracle category). The CLI exit code is `68`, the MCP `kind` is `idempotency_conflict`, the SDK throws `HexalithFoldersApiException<ProblemDetails>` with `problem.Category` resolving to `idempotency_conflict` via `[EnumMember]`, REST returns `409` with canonical RFC 9457 + `category == "idempotency_conflict"`. Source-gate per Story 5.6 pattern on the oracle row's `outcome_mapping` for `idempotency_conflict`.
