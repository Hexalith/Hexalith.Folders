# Epic 10 Retrospective: Folders Worker-Side Semantic-Indexing Producer And Bridge Projection

## Epic Summary

Epic 10 completed the second phase of the Memories search-index integration track. It turned the Epic 9 dormant `hexalith-folders -> folders-index` routing into a Folders-owned search-index pipeline:

- Story 10.1 created the worker-side indexing port and dependency boundary.
- Story 10.2 added the Folders-owned bridge projection for indexed, stale, skipped, failed, tombstoned, and reconciliation-required status.
- Story 10.3 connected durable file-write and commit evidence to `SearchIndexEntryChanged` publication through Dapr pub/sub.
- Story 10.4 added hard removal and soft archive semantics through `SearchIndexEntryRemoved` and `SearchIndexEntryChanged{folders.status=archived}`.
- Story 10.5 exposed the authorized Folders query facade over the shared `folders-index` Memories tenant, with API, SDK, MCP, CLI, PRD, architecture, and console-status coverage.

Sprint status shows 5 of 5 Epic 10 stories `done`; this retrospective closes `epic-10-retrospective` and marks Epic 10 complete from a story-tracking perspective.

## What Went Well

- The epic delivered a coherent end-to-end slice instead of isolated plumbing. By the end of Story 10.5, producer, bridge state, removal/archive behavior, query facade, contract parity, and console visibility all aligned around the same `folders-index` model.
- The mechanism correction was handled transparently. Story 10.3 originally targeted `MemoriesClient.IngestAsync`, then the team identified that this was the separate RAG ingestion subsystem and reworked the story to publish `SearchIndexEntryChanged` CloudEvents to `pubsub` / `memories-events`.
- Review caught load-bearing defects before they hardened into contracts. The folder-level `AggregateId` data-loss issue, inbound `folders.events` topic mismatch, invalid-payload redelivery behavior, archive-after-hard-delete resurrection, and query-facade pagination/trim issues were all fixed through adversarial review.
- The Folders-owned bridge projection became the right source of local indexing truth. It tracks status without treating Memories as authoritative Folders state, preserves metadata-only evidence, and prevents older indexing outcomes from overwriting newer file versions.
- The shared-index security model was explicit. Story 10.5 correctly treats `folders-index` as a shared physical Memories tenant and applies Folders-side tenant, organization, folder, and workspace trimming before returning any result.
- The public contract moved forward with the product. FR58, the architecture Query Facade section, OpenAPI operations, SDK generation, MCP tools, CLI command, and parity fixtures were updated together.
- The team preserved the metadata-only boundary. The facade drops snippets and source URIs, producer attributes are curated, and sentinel-corpus style checks were added or hardened during review.

## Challenges And Growth Areas

- The epic repeatedly proved that green structural tests can still miss runtime wiring defects. Story 10.3 had a passing suite while the real EventStore-to-worker topic binding was wrong. The fix was not just a code patch; it required production override, pub/sub scope, conformance coverage, and a Tier-3 harness.
- Live `aspire run` evidence remains blocked by the DCP-capable lane. Story 10.4's AC9 and Story 10.5's live facade sign-off still rely on a real AppHost round-trip that local artifacts cannot prove.
- The real mutation-to-index path is still not fully populated by a production content materializer. Story 10.4's gated live test seeds through the worker port to prove indexing mechanics; a deeper folder-mutation-to-content-materialization-to-index proof remains tracked.
- Story 10.5 intentionally accepted a deferred live Server bridge-read wiring gap. The query facade is safe and fails closed, but the deployed Server facade can remain inert until the EventStore-backed bridge read model is wired or the deferral is explicitly carried into release readiness.
- The Dapr invoke allow-rule for the Memories read facade is conditional because `AddMemoriesClient` uses a direct base-address `HttpClient`. Architecture now documents that the Dapr policy is operative only when the base address routes through the sidecar, but this is still easy to over-claim in release evidence.
- The `.slnx` canonical inventory drift resurfaced in both Story 10.4 and 10.5 verification. It was pre-existing relative to those changes, but it weakens confidence in a required testing lane until reconciled.
- The naming remains misleading. "SemanticIndexing" is retained for traceability, but the implementation is the syntactic/BM25 Memories search index, not the RAG memory-ingestion path.

## Key Lessons

1. A search-index integration is not the same thing as a RAG ingestion integration. The implementation mechanism must be proven from the receiving service's actual contract, not inferred from similarly named client APIs.
2. Pub/sub wiring needs live or cross-process evidence. Tests that call handlers directly or assert declared metadata are useful, but they do not prove that events can travel through Dapr and AppHost resource wiring.
3. Search-index identity grain is a correctness boundary. Folder-level upsert keys would silently overwrite sibling file versions; per-file-version `AggregateId` and stable source URI are load-bearing invariants.
4. Delete and archive semantics need product decisions before implementation. Epic 10's hybrid model is now explicit: file removal is hard delete; folder archive is soft delete with `folders.status=archived`.
5. A shared physical index must be treated as untrusted. Memories can rank candidate rows; Folders must authorize, trim, hydrate, and redact before anything crosses the public boundary.
6. Fail-safe defaults should be visible as unavailable, not healthy empty. Story 10.5's indexing-status patches moved this in the right direction and should be the pattern for future read models.
7. Names matter in long-lived architecture. Keeping "semantic" labels while implementing syntactic/BM25 search-indexing is understandable for traceability, but it should be resolved or explicitly documented.

## Significant Discoveries

- The original Story 10.3 `IngestAsync` direction was wrong for the Epic 9 routing. The correct path is `SearchIndexEntryChanged` / `SearchIndexEntryRemoved` CloudEvents over Dapr pub/sub to `memories-events`.
- The inbound `folders.events` topic needed an EventStore publisher topic override. Without that, managed-tenant folder events would not reach the worker subscription reliably.
- Story 10.4 discovered that Memories upsert is destructive. Archive soft delete must resend the full document with `folders.status=archived`, not patch a single field.
- Story 10.5 discovered and documented the facade egress mismatch: the Dapr invoke policy exists, but the typed Memories client is direct-base-address unless configured through the sidecar route.
- No Epic 11 is defined in `epics.md`. Preparation should focus on release-readiness closure and explicit follow-up ownership rather than assuming the next numbered epic exists.
