## What Went Well

- **Context-rich, code-grounded story specs converted risky infra work into first-pass-green tasks.** Every story carried exhaustive Dev Notes: the exact current-state code being replaced, the canonical Tenants AppHost target pattern, pinned helper signatures, line-level source references, and a "Critical Implementation Notes" block that pre-identified the highest-risk traps **before any code was written**. Story 9.1's spec pre-discovered that the platform helper would `ArgumentNullException` on `adminServer: null` (`HexalithEventStoreExtensions.cs:141`) and laid out Option A vs Option B with full evidence — turning a cross-submodule change into a deterministic decision rather than a mid-implementation surprise.
- **The cross-submodule change was made backward-compatible and proven so.** 9.1's Option A made `adminServer`/`adminUI` nullable and guarded all admin/UI wiring behind `if (adminServer is not null)`, then **verified** backward-compatibility by building `Hexalith.EventStore.AppHost` (the canonical non-null caller) against the modified helper. The change benefits every platform consumer, not just Folders, and broke no existing caller.
- **Closed-set conformance tests made the correct change the only green change.** `AppHostPlatformCompositionConformanceTests`, `DaprPolicyConformanceTests`, and `ContainerImageConformanceTests` derive their expectations from closed sets keyed off `FoldersAspireModule` constants. Adding `memories` (9.2) forced exactly: move `memories` forbidden→expected scope, add to `StableAppIds`, add the sixth deny-by-default doc + sidecar binding + empty pub/sub scopes, and regenerate the semantic hash — and nothing else passed. The guardrails caught scope drift precisely instead of relying on reviewer vigilance.
- **Deny-by-default honesty discipline was preserved (continuity from Epic 7).** 9.2 registered `memories` with `policies: []` and empty pub/sub topic scopes, explicitly declining to invent speculative `folders → memories` invoke rules that the negative-test suite could not meaningfully exercise. 9.3 added routing env vars with **zero** policy/deploy changes (routing ≠ permission). This is exactly the Epic 7 "reference-pending / no narrated-coverage" posture applied correctly.
- **Adversarial review added real coverage, not nits.** Each Senior Developer Review independently re-ran the build and the three suites (not trusting reported counts) and auto-fixed genuine gaps: 9.1 added a test exercising the production **YAML-`LocalPath`** component-sourcing branch (the existing topology tests only drove the in-code/null-path branch); 9.3 added a null-guard `[Fact]` for the new helper plus a return-contract `ShouldBeSameAs` assertion. Test counts rose because the additions were real and green (605→610 on IntegrationTests across the epic).
- **Non-circular test design was a conscious choice.** 9.3 encapsulated the routing in a single production helper (`WithFoldersMemoriesSourceRouting`) that **both** `Program.cs` and the test invoke, deliberately avoiding the "re-implement the wiring in the test and then assert it" circular trap. The test exercises the same code path production uses; the constant assertions are the guaranteed-green contract floor.
- **Strict scope phasing held cleanly across all three stories.** Each story named exactly what it deferred and to whom: 9.1→9.2 (no `memories`), 9.2→9.3 (no routing), 9.3→Epic 10 (no producer, no invoke-auth, no pub/sub topic). No story over-reached its boundary, and the handoff doc records the Epic 10 line precisely. This is what kept a multi-store infra change reviewable in independent increments.

## Challenges And Growth Areas

- **`aspire run` live boot was never verified — across all three stories.** An **environment-wide Aspire CLI (13.4.5) vs DCP/Hosting (13.4.6) `--tls-cert-file` certificate-trust mismatch** aborted DCP orchestration before any resource was created, in 9.1, 9.2, and 9.3 identically. The live-boot half of AC6 (9.1) / AC7 (9.2, 9.3) is therefore **unverified**; composition correctness rests entirely on build + structural/conformance tests. This is the single largest residual gap: the composed topology — `eventstore`/`tenants`/`memories`(+`memories-vectors`/`memories-graphs`)/`folders`/`folders-workers`/`folders-ui`, and the `folders-index` auto-provisioning — has never actually been observed booting healthy. It is an environmental toolchain defect, **not** a topology defect, but an entire AC half is owed to a DCP-capable environment / CI.
- **Three-way Aspire version drift in planning artifacts was flagged but left unreconciled.** Verified during this retro: `Directory.Packages.props` (authoritative) pins **Aspire 13.4.6** + `CommunityToolkit.Aspire.Hosting.Dapr 13.4.0-preview.1.260602-0230`, while `architecture.md` (§420, §561/I-1, §1488) still says **13.3.0** + 13.0.0, and both `epics.md` AR-INFRA-01 (§328) and `project-context.md` (§41) say **13.4.3** + 13.0.0. The sprint-change-proposal listed a §1486 version-note reconciliation as Phase-1 doc-sync (§4.2), so it was technically in Epic 9's scope, but 9.1 and 9.3 both deliberately punted it as "a focused follow-up against the authoritative `Directory.Packages.props`." It remains open. *(Reconciled in this retro's documentation pass — see Action Items + the doc-update appendix.)*
- **Cross-repo package-version coupling surfaced mid-implementation (NU1605).** 9.1's restore failed because Folders pinned `CommunityToolkit.Aspire.Hosting.Dapr 13.0.0` while the platform `.Aspire` helpers pulled 13.4.0-preview — a transitive downgrade. Resolved (user-approved) by aligning the Folders central pin. As more `.Aspire` helpers are consumed cross-repo, this version-coupling will keep recurring unless tracked deliberately.
- **Transient `bin`/`obj` build file-lock races recurred (MSB3026/MSB4018).** Caused by the VS Code C# Dev Kit build host locking output dirs while `dotnet build` ran concurrently — observed in 9.1 and 9.2, clears on a clean retry. Environmental, not a code defect, but it injects noise into "0 Warning(s)" claims and forced retries.
- **The checked-in `resiliency.yaml` is functionally inert in gateway-only mode (9.1 LOW-1).** The platform helper consumes `resiliencyConfigPath` **only** to set the admin-server's `AdminServer__ResiliencyConfigPath` env var; with `adminServer: null`, that block is skipped, so `resiliency.yaml` is passed but wired into no sidecar's resources. It is behavior-preserving (the original Folders topology had no resiliency at all) and correctly **not** auto-fixed inside a behavior-preserving refactor — but it is dead config that reads as live, and is owed a resolution (helper enhancement or removal-with-rationale).
- **Minor dev-record test-count slips recurred (caught in review).** 9.2 reported 268/606 vs actual 274/608; 9.3 reported 609 vs actual 610. In every case reality was strictly *better* (devs added more tests than the minimum and under-counted them), and review corrected the records — but the pattern echoes the Epic 7 "record accuracy slipped" lesson: reconcile counts at story close, not only at review.

## Key Lessons

1. **Front-loading exhaustive, code-grounded story specs is the highest-leverage practice for infra refactors.** Current-state code + canonical target pattern + pinned signatures + pre-identified traps ("Critical Implementation Notes") is what made a cross-submodule change a first-pass-green task. Pre-discovering the `adminServer: null` throw and the closed-set conformance traps removed the two biggest sources of mid-flight churn.
2. **Closed-set conformance tests are a double-edged contract.** They make the correct change the only green change (an excellent guardrail), but they also mean every additive change must touch a precise, enumerated artifact set in lockstep — hash regen, `StableAppIds`, scope lists, sidecar bindings. Document that exact lockstep set in the story (9.2 did this well in its Critical Implementation Notes) so the dev edits the whole set, not a subset.
3. **Honest deny-by-default beats speculative permissions for not-yet-callable services.** `memories` ships with no callers and empty topic scopes because no producer exists; inventing `folders → memories` allow-rules now would create conformance cases the negative suite cannot meaningfully exercise. The real caller policies + 7-category negative rows land in Epic 10 with the producer.
4. **Encapsulate config-wiring in a production helper that both Program and tests call.** A test that re-implements the wiring inline and asserts it proves nothing about production. The `WithFoldersMemoriesSourceRouting` pattern gives non-circular coverage; pair it with constant assertions as the guaranteed-green floor.
5. **An entire AC half resting on an environmental blocker is a real, named gap — not "done."** Structural/conformance tests prove composition correctness but do **not** prove the topology boots. A DCP-capable boot lane (or `aspire update` to align CLI↔DCP) is required to actually close AC6/AC7's live-boot half.
6. **Cross-repo version coupling needs proactive tracking, not reactive NU1605 firefighting.** When a repo consumes sibling `.Aspire` helpers under separate central package management, transitive-downgrade conflicts are predictable. Align the shared pins deliberately and document the coupling.
