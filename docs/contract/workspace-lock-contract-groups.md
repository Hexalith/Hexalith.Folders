# Workspace And Lock Contract Groups

Story 1.8 adds only Contract Spine declarations for workspace preparation, workspace lock acquisition, lock release, lock inspection, retry eligibility, and C6 transition evidence.

The canonical machine-readable source remains `src/Hexalith.Folders.Contracts/openapi/hexalith.folders.v1.yaml`. These notes record operation mapping, the authorization-first response contract, and deferred decisions only; they do not define runtime behavior.

## Authorization-First Response Contract

For all workspace/lock operations, authentication and authorization are evaluated before any state-specific outcome is surfaced:

- Unauthenticated callers receive the canonical 401 `SafeAuthorizationDenial401` envelope.
- Authenticated callers without tenant access, folder ACL, workspace access, lock-scope access, or task scope receive 403 or 404 `SafeAuthorizationDenial` envelopes that are externally indistinguishable across absent, cross-tenant, missing-binding, missing-policy, missing-workspace, missing-lock, missing-task, and stale-read cases.
- Distinguishable responses such as 410 `WorkspaceLockExpired`, 409 `WorkspaceLockConflict`, 409 `WorkspaceAuthorizationRevoked`, 422 `WorkspaceTransitionInvalid`, 503 `ProviderUnavailable`, 503 `UnknownProviderOutcome`, and lock or lease metadata in `WorkspaceLockStatus` are only reachable after authorization for the requested scope succeeds. They therefore do not leak resource existence, lease history, lock identity, or provider state to unauthorized callers.

This is the safe-denial discipline AC2/AC11 requires. Operation-level Problem Details examples include authorization-first preamble in their summaries so SDK consumers cannot mistake the response for an unauthenticated path.

## Workspace and Lock Schema Notes

- `WorkspaceLockStatus.lockState` uses the narrow `LockState` enum (`unlocked`, `locked`, `expired`, `stale`, `revoked`) — distinct from the full workspace `LifecycleState` so a lock cannot accidentally surface a non-lock state.
- `WorkspaceLockStatus.lease` is optional. Workspaces that have never been locked or that are currently unlocked omit the lease block entirely.
- `WorkspaceRetryEligibility.taskId` is optional so workspace-scoped lock inspection does not have to carry a task scope; mutating commands still propagate `X-Hexalith-Task-Id`.
- `WorkspaceTransitionEvidence` exposes a `lockEvidence` sub-shape (`LockLeaseMetadata`) for transitions that affect the workspace lock (e.g. `WorkspaceLocked`, `WorkspaceLockReleased`, `LockLeaseExpired`), satisfying AC15's request for lock identity, lease status, acquired/effective/expiry timestamps, and authorized-caller holder reference.
- `WorkspaceTransitionAttempt.toState` is required so attempted transitions are always fully described.
- `releaseReasonCode` includes `task_cancelled` and `lock_revoked` so SDK callers can audit those release paths without collapsing them into `caller_abandoned`.

## Deferred Decisions

- Exact retry policy, default lease duration, renewal policy, clock skew policy, expanded audit taxonomy, runtime authorization-revocation mechanics, and reconciliation execution remain deferred to Epic 4 and later operational stories.
- C3 retention values remain proposed workshop values until Legal and PM approval. Story 1.8 uses mutation-tier idempotency only and does not bind commit retention.
- C4 input limits remain proposed workshop values until PM approval. Workspace and lock responses are metadata-only and do not introduce path, file, or context-query limits.
- C6 vocabulary is consumed from `docs/exit-criteria/c6-transition-matrix-mapping.md`; this story exposes evidence shapes and the `state_transition_invalid` category without implementing the aggregate matrix.

## Lock Ownership Proof

`lockOwnershipProof` is a non-secret opaque lock proof scoped to the authenticated tenant, folder, workspace, task, and lock. It is not an authentication credential, provider credential, session secret, refresh credential, bearer authorization value, or authorization override. Unauthorized Problem Details and audit diagnostics must omit the proof value; contract tests verify the proof value never appears in `SafeAuthorizationDenial*`, `WorkspaceLockExpiredProblem`, `WorkspaceLockConflictProblem`, `WorkspaceAuthorizationRevokedProblem`, `WorkspaceTransitionInvalidProblem`, `ProviderReadinessFailedProblem`, `WorkspacePreparationFailedProblem`, `UnknownProviderOutcomeProblem`, `ProviderUnavailableProblem`, `IdempotencyConflictGeneric`, or `ReconciliationRequiredProblem` examples.
