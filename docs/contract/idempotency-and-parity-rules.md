# Idempotency and Adapter Parity Rules

status: finalized MVP rules with deferred implementation owners
source inputs: PRD command/query contract, PRD error codes, architecture Adapter Parity Contract, C3 retention artifact, C4 input limits artifact, Story 1.3 fixture seeds, Story 1.4 Phase 0.5 deliverables
last reviewed: 2026-05-11

This document is the implementation-facing contract-rules artifact for Story 1.5. It defines operation metadata, idempotency equivalence, non-mutating read consistency, and adapter behavioral parity before the OpenAPI Contract Spine is authored.

It does not author endpoint paths, OpenAPI schemas, SDK methods, CLI commands, MCP tools, REST handlers, generated helpers, parity result rows, idempotency persistence, or domain behavior.

## Decision Record

Finalized MVP rules:

- Tenant-scoped idempotency equivalence is mandatory for every mutating command. Equivalent replay requires the same authoritative tenant, same operation intent, same semantic request, same target resource identity, same behavior-affecting credential scope, and same idempotency key.
- Mutating commands require `Idempotency-Key`. Same key plus equivalent payload returns the same logical result. Same key plus different payload returns canonical `idempotency_conflict`.
- Non-mutating operations do not accept `Idempotency-Key`; they still require correlation behavior, authorization parity, safe denial shape, audit metadata, and read-consistency classification.
- SDK is the typed canonical client. REST and SDK own transport parity. CLI and MCP are behavioral wrappers that keep pre-SDK validation, post-SDK error projection, idempotency-key sourcing, correlation sourcing, task ID sourcing, credential sourcing, CLI exit codes, and MCP failure kinds testable.
- Examples and fixtures are metadata-only. They must not include file contents, diffs, provider tokens, credential material, raw provider payloads, production URLs, generated context payloads, or unauthorized resource existence.
- `Hexalith.Folders.Contracts` remains behavior-free. Rules and schemas may describe future generation behavior, but generated `ComputeIdempotencyHash()` helpers and runtime persistence belong to later stories.

Deferred work:

- Story 1.6 owns OpenAPI 3.1 path, schema, header, Problem Details, and shared extension vocabulary shape.
- Story 1.12 owns generated SDK method signatures, NSwag wiring, and generated `ComputeIdempotencyHash()` helpers.
- Stories 1.7 through 1.11 own operation-group Contract Spine details.
- Story 1.13 owns final `tests/fixtures/parity-contract.yaml` row generation and C13 oracle consumption.
- Epic 4 owns runtime idempotency persistence, workspace lifecycle behavior, provider side effects, and reconciliation behavior.
- Epic 5 owns final SDK, CLI, and MCP implementations and behavioral parity tests.

Human-decision state:

- C3 retention values are proposed workshop values and need Legal + PM approval before Story 1.6 treats them as binding Contract Spine defaults. Commit idempotency TTL references the proposed C3 audit metadata retention class and remains approval-dependent.
- C4 input limits are proposed workshop values and need PM approval before Story 1.6 writes binding schema limits. This document records the C4 classes and bounds as the current Phase 0.5 input, not as implementation code.
- Operation IDs for Phase 1 inventory candidates are stable planning names only until Story 1.6+ freezes exact OpenAPI operation IDs.

Parser-policy classification for future canonicalization:

| input_case | classification | rationale | deferred_owner |
|---|---|---|---|
| duplicate JSON keys | parser-rejected | Ambiguous semantic payload; adapters must not guess which duplicate key wins. | Story 1.6 schema/error mapping |
| null versus omitted values | non-equivalent unless schema declares default equivalence | Null is an explicit semantic value; omitted is absent input. | Story 1.12 canonical helper generation |
| percent-encoded path equivalence | equivalent only after a single safe decode maps to the same normalized path metadata | Prevents encoded traversal and double-decode ambiguity. | Story 1.9 path contracts and Story 1.12 helpers |
| Unicode normalization form | equivalent only through generated helper normalization for fields declared normalization-insensitive | Callers and adapters must not hand-roll Unicode canonicalization. | Story 1.12 helpers |
| casing-sensitive identifiers | non-equivalent by default | Tenant IDs, folder IDs, operation IDs, task IDs, lock tokens, and opaque provider references preserve declared casing. | Story 1.6 schema vocabulary |
| ULID casing | equivalent only for fields explicitly declared ULID-typed | Crockford Base32 casing may normalize for typed ULIDs, not arbitrary strings. | Story 1.12 helpers |

## Canonical Correlation Field Path Per Surface

The schema's `correlation_field_path` regex permits any of `headers|problem|result|metadata` prefixes for forward-compat. Adapters and parity-row generators must pick the canonical prefix per surface from this table. Mixing surfaces is permitted only when the same operation is observed twice (e.g., REST receives `headers.X-Hexalith-Correlation-Id` and the SDK projects the same id into `metadata.correlation_id`).

| surface | success_path | error_path | rationale |
|---|---|---|---|
| REST request | `headers.X-Hexalith-Correlation-Id` | `headers.X-Hexalith-Correlation-Id` | Wire-format correlation; client controls echo. |
| REST response success | `headers.X-Hexalith-Correlation-Id` | n/a | Server echoes caller value or returns generated value. |
| REST response error | n/a | `problem.correlation_id` | RFC 9457 Problem Details extension member. |
| SDK | `result.metadata.correlation_id` | `metadata.correlation_id` (exception) | Typed result and exception both expose the id. |
| CLI success | `metadata.correlation_id` (structured output) | n/a | CLI structured output exposes the id; defaults to invocation echo on stderr. |
| CLI error | n/a | `problem.correlation_id` | CLI prints canonical Problem Details for failures. |
| MCP | `metadata.correlation_id` | `metadata.correlation_id` | MCP tool envelopes always include metadata; success or failure path is the same. |
