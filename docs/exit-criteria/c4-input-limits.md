## Rationale

The MVP should fail predictably before a query can become an unbounded tenant or repository scan. Bounds apply before or after authorization according to the table so implementers cannot search first and filter later. The values are conservative enough for early consumers while keeping C1 and C5 capacity calibration separate.

The proposed rows are global MVP defaults. Tenant-tunable limits are deferred until release calibration has evidence and a configuration model.

## Verification impact

Verification must prove every numeric value has units, boundary semantics, provenance, approval state, a future consuming artifact, and a review date. Story 1.6 must carry these values into OpenAPI keywords or Hexalith extension metadata without creating per-tenant override files.

## Deferred implementation

This document does not create or edit OpenAPI, SDK generation, query handlers, middleware, rate limiters, tenant configuration, operations-console UI, or CI gates. PM approval remains the blocker before Contract Spine authoring treats these values as binding defaults.
